Notifications are one-way MCP messages. Use them to tell connected clients that something changed, report progress, or send custom status events.

Broadcast notifications, session-targeted notifications, and resource-subscription updates require stateful sessions. Request-scoped progress and log notifications depend on the current tool call and the client-provided request context. In stateless edge environments, do not design workflows that depend on server-to-client session notifications.

This guide focuses on notification workflows. Use the [MCPServer API reference](/typescript/api-reference/server/mcp-server) and [Tool context API reference](/typescript/api-reference/server/tool-context) for exact method signatures, return values, and payload shapes.

## Choose the notification path

Choose the narrowest target that matches the workflow.

| Need | Use |
| --- | --- |
| Notify every connected client | `server.sendNotification(...)` |
| Notify one known session | `server.sendNotificationToSession(...)` |
| Notify the client running the current tool | `ctx.sendNotification(...)` |
| Report progress for the current tool call | `ctx.reportProgress?.(...)` |
| Tell clients to refresh tools, resources, or prompts | list-changed helpers |
| Tell subscribed clients a resource changed | `server.notifyResourceUpdated(uri)` |

Use resource subscriptions for resource-specific updates. Use custom notifications for status events that are not tied to one resource.

## Send a custom notification

Use a namespaced method for custom notifications so clients can route them cleanly.

```typescript
await server.sendNotification("custom/status-update", {
  status: "ready",
  checkedAt: new Date().toISOString(),
});
```

Custom notifications are best for dashboards, background job status, and operational events. Keep payloads small and stable.

## Notify the current client from a tool

Inside a tool callback, use `ctx.sendNotification(...)` when only the calling client should receive the message.

```typescript
import { text } from "mcp-use/server";

server.tool(
  {
    name: "run_import",
    description: "Run an import and notify the caller when it starts.",
  },
  async (_params, ctx) => {
    await ctx.sendNotification("custom/import-started", {
      startedAt: new Date().toISOString(),
    });

    await runImport();
    return text("Import complete.");
  },
);
```

Use this for per-user status that should not be broadcast to other sessions.

## Notify a specific session

Use session IDs when another operation needs to target one connected client.

```typescript
const sessions = server.getActiveSessions();
const [sessionId] = sessions;

if (sessionId) {
  const sent = await server.sendNotificationToSession(sessionId, "custom/welcome", {
    message: "Connected.",
  });

  if (!sent) {
    console.log("Session was no longer available.");
  }
}
```

Session-targeted notifications are only useful while the session remains connected. Use persistent storage for durable tasks that must survive disconnects.

## Refresh client lists after dynamic changes

If your server adds or removes tools, resources, or prompts at runtime, notify clients to refresh their cached lists.

```typescript
await server.sendToolsListChanged();
await server.sendResourcesListChanged();
await server.sendPromptsListChanged();
```

Call these after the registration set changes. Do not call them for normal tool results or ordinary resource content updates.

## Report progress from long-running tools

Use `ctx.reportProgress` when the current tool call has stages the client should see.

```typescript
server.tool(
  {
    name: "process_data",
    description: "Process data in stages.",
  },
  async (_params, ctx) => {
    await ctx.reportProgress?.(0, 100, "Starting");
    await processStageOne();
    await ctx.reportProgress?.(50, 100, "Halfway");
    await processStageTwo();
    await ctx.reportProgress?.(100, 100, "Complete");
    return text("Done.");
  },
);
```

`ctx.reportProgress` is optional because clients only provide progress tokens for some calls. Always use optional chaining or a guard.

## Test notifications locally

Use a stateful local connection and a client that displays notifications.

```bash
cd libraries/typescript/packages/mcp-use/examples/server/features/notifications
pnpm install
pnpm dev
```

In the Inspector, call tools that send custom notifications. If you use the example `toggle-mode` tool, confirm that the client receives a tools-list-changed notification.
