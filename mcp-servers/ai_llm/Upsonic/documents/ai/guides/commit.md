# Commit Rules

## Hard Rule — Never Commit Without Approval

Do **not** run `git commit`, `git push`, or any history-rewriting command (`reset --hard`, `rebase`, `amend`, force-push) until the human has reviewed the diff and explicitly said to commit.

When work looks done, stop and say "ready to commit, want me to?". Wait for an explicit "yes" / "commit it" / equivalent. Silence is not approval.

## Format

```
type(scope): subject
```

- **type**: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf`, `style`, `build`, `ci`
- **scope**: the touched area (`agent`, `prebuilt`, `models`, `tools`, …) — omit if it spans many
- **subject**: imperative, lowercase, no trailing period, ≤ 72 chars

Match repo style:
```
refactor(prebuilt): standardize layout under src/upsonic/prebuilt
feat(agent): add cost property aggregating tokens across tasks
fix: fix conflicts between PRs
chore: move uv to dev dependencies
test(agent): add smoke tests for Agent.cost accumulation
```

## Headers — Keep Them Tight

- One line. No emoji. No "fixes #123" in the header — put issue refs in the body.
- Say *what* changed, not *how*. "add retry to embeddings" beats "loop through embeddings calls with try/except".
- No version bumps, dates, or author tags in the header.

## Body (Optional, Only If Needed)

- Blank line after the subject, then the body.
- Wrap at 72 chars.
- Explain **why** the change is needed when it isn't obvious. Skip the body for trivial changes.
- Bullet points are fine; full paragraphs are fine. Do not narrate every file.
- Reference issues/PRs in the body: `Refs #123`, `Closes #456`.

## Don't

- Don't commit secrets (`.env`, credentials, tokens) — warn the user if they ask.
- Don't `git add -A` blindly. Stage specific files.
- Don't bypass hooks (`--no-verify`, `--no-gpg-sign`) unless the user says so.
- Don't amend or rebase published commits.
- Don't write multi-paragraph essays in the subject line.
- Don't include "Generated by …" / Co-Authored-By unless the user asks.

## Workflow

1. Make changes.
2. `git status` + `git diff` — review what's staged.
3. Show the diff summary to the user.
4. Propose a commit message.
5. **Wait for explicit approval.**
6. Stage the specific files and commit.
7. Run `git status` after the commit to confirm clean state.
8. Do not push unless the user asks.
